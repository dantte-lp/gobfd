# Реестр рисков зависимостей

Документ отслеживает внешние зависимости с нетривиальным риском из-за статуса сопровождения, проектов с единственным мейнтейнером или изменений в upstream. Каждая запись включает оценку риска и стратегию митигации.

## reeflective/console — Средний риск

| Поле | Значение |
|------|----------|
| **Модуль** | `github.com/reeflective/console v0.1.25` |
| **Используется в** | `cmd/gobfdctl/commands/shell.go` — интерактивная REPL-оболочка |
| **Уровень риска** | Средний |
| **Причина** | Библиотека до v1.0, единственный мейнтейнер, 6 транзитивных зависимостей |

### Описание

Библиотека `reeflective/console` обеспечивает интерактивную оболочку для `gobfdctl`. Она оборачивает команды Cobra в readline-based REPL с табуляционным дополнением, историей и настройкой приглашения.

### Факторы риска

- **Pre-1.0 API**: Библиотека не достигла v1.0, что означает возможные ломающие изменения между минорными версиями.
- **Единственный мейнтейнер**: Проект сопровождается одним разработчиком. Bus factor = 1.
- **Транзитивные зависимости**: Тянет 6 дополнительных зависимостей (readline, completions и др.).
- **Нет альтернативы в stdlib**: Стандартная библиотека Go не включает Cobra-совместимый REPL-фреймворк.

### Митигация

1. **Фиксация версии**: Зависимость зафиксирована на `v0.1.25` в `go.mod`. Обновления должны тестироваться перед принятием.
2. **Вендоринг**: Если проект станет заброшенным, вендорить исходный код в `internal/console/`.
3. **Graceful degradation**: Неинтерактивный CLI (`gobfdctl <command>`) работает без этой зависимости. Интерактивная оболочка — вспомогательная функция.
4. **Ограниченная область**: Зависимость изолирована в одном файле (`shell.go`) и не затрагивает демон или реализацию протокола.

## osrg/gobgp/v3 — Высокий риск

| Поле | Значение |
|------|----------|
| **Модуль** | `github.com/osrg/gobgp/v3 v3.37.0` |
| **Используется в** | `internal/gobgp/` — опциональная интеграция GoBGP |
| **Известный advisory** | `GO-2026-4736` / `CVE-2026-30405` / `GHSA-4p9m-8gc4-rw2h` |
| **Уровень риска** | Высокий |

### Описание

GoBGP затронут denial-of-service advisory в обработке BGP path attribute
NEXT_HOP. По состоянию на 2026-04-22 база Go vulnerability database не указывает
исправленную версию.

### Митигация

1. **Ограниченная экспозиция**: путь GoBGP является опциональным и должен
   подключаться только к GoBGP gRPC endpoint на localhost или в доверенной
   management-сети.
2. **Контролируемый CI allowlist**: `scripts/vuln-audit.go` разрешает только
   `GO-2026-4736`; любой дополнительный finding из `govulncheck` или OSV ломает
   CI.
3. **Триггер обновления**: удалить allowlist-запись после публикации
   исправленного релиза GoBGP и обновления модуля через `go mod tidy`.

## gopkg.in/yaml.v3 — Низкий риск

| Поле | Значение |
|------|----------|
| **Модуль** | `gopkg.in/yaml.v3 v3.0.1` (прямая) |
| **Наследник** | `go.yaml.in/yaml/v3 v3.0.4` (уже непрямая зависимость) |
| **Используется** | `koanf/parsers/yaml` — парсинг YAML-конфигурации |
| **Уровень риска** | Низкий |

### Описание

Модуль `gopkg.in/yaml.v3` был архивирован мейнтейнером в апреле 2025 года. Модуль-наследник `go.yaml.in/yaml/v3` совместим по API и уже присутствует как непрямая зависимость через другие пакеты.

### Факторы риска

- **Архивированный репозиторий**: Дальнейшие исправления ошибок и патчи безопасности для пути `gopkg.in` выпускаться не будут.
- **Миграция upstream в ожидании**: GoBFD использует yaml.v3 через `koanf/parsers/yaml`. Мейнтейнеры koanf пока не мигрировали на путь наследника.

### Митигация

1. **Изменения кода не нужны**: Наследник `go.yaml.in/yaml/v3` совместим по API. Когда koanf мигрирует, `go.mod` GoBFD автоматически переключится при `go mod tidy`.
2. **Уже непрямая**: GoBFD не импортирует yaml.v3 напрямую в коде приложения — он проходит через парсер конфигурации koanf.
3. **Мониторинг upstream**: Отслеживать issue koanf по миграции yaml.v3. При решении выполнить `go mod tidy` для завершения переключения.
4. **Нет проблем безопасности**: Архивированный модуль `v3.0.1` не имеет известных уязвимостей. Наследник сопровождается тем же автором под новым путём модуля.
