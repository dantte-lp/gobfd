## 1. osrg/gobgp#2563 — Native BFD support (OPEN)

> Background context: `@fujita` already endorsed the
> "separate BFD daemon over gRPC" pattern in
> [#1974780853](https://github.com/osrg/gobgp/issues/2563#issuecomment-1974780853).
> `@dylandreimerink`'s rationale matches the GoBFD design exactly.

```
Quoting @fujita upthread:

> If your BFD implementation needs a new gRPC API to manage GoBGP, feel free to make a pull request.

@dylandreimerink's rationale (the daemon should run on the forwarding
device, BFD is protocol-agnostic, decouple from BGP control plane) is
the exact reason I have been building **GoBFD** — a Go-native BFD
daemon at https://github.com/dantte-lp/gobfd. Disclosure: I'm the
author.

`v0.6.0` (just tagged) ships:

- RFC 5880/5881/5882/5883/7419/9384/9468/9747/9764 implemented
- Typed gRPC services per session family: `BfdService`, `EchoService`
  (RFC 9747), `MicroBFDService` (RFC 7130). All exposed through
  ConnectRPC over HTTP/2 plaintext.
- Built-in `gobgp.Client` integration that drives `DisablePeer` /
  `EnablePeer` on BFD state changes with RFC 5882 §3.2 flap dampening.
  Runnable demo at
  `deployments/integrations/bgp-fast-failover/`.

I see #3388 is now adding partial in-process BFD. Both directions are
useful: embedded for routing-only deployments, standalone for cases
where BFD also needs to feed OSPF/IS-IS/loadbalancers, where the data
plane should survive a GoBGP restart, or where you need RFC 9747 echo /
RFC 8971 VXLAN / RFC 9521 Geneve. If a small additional gRPC verb on
GoBGP would help the standalone path (e.g. richer state-change
notifications), happy to draft a PR.
```

## 3. siderolabs/talos#11126 — Native GoBGP integration (OPEN)

> Background context: `@maxpain` (the author) explicitly named lack of
> BFD as the main GoBGP gap. Direct fit.

```
@maxpain raised that the main GoBGP gap for this use case is BFD. As a
data point: I've been building **GoBFD**
(https://github.com/dantte-lp/gobfd) — a Go-native BFD daemon designed
exactly for the "BFD as a sibling process to GoBGP" pattern. It
watches sessions and drives `DisablePeer` / `EnablePeer` over GoBGP's
gRPC API on state changes, implementing RFC 5882 §3.2 flap dampening.

`v0.6.0` (just tagged) adds typed gRPC services for RFC 9747 echo and
RFC 7130 micro-BFD on top of the base session family.
`deployments/integrations/bgp-fast-failover/` is a runnable demo.
`deployments/integrations/kubernetes/` is a Kubernetes DaemonSet
example.

This doesn't replace what #11126 is asking for (native MachineConfig
integration). It's a candidate for a Talos system extension that keeps
BFD decoupled from the BGP control plane — which @ecbaldwin argues for
in [osrg/gobgp#2006](https://github.com/osrg/gobgp/issues/2006#issuecomment-845307627)
(graceful-restart compatibility). Happy to discuss what
Talos-friendly packaging would look like if useful.

Disclosure: I'm the author.
```

## Posting checklist

Before posting any of the above:

- [ ] Confirm the GoBFD release tag in the comment matches the
      latest tag on `master`.
- [ ] Confirm referenced upstream PR / issue numbers are still valid.
- [ ] Verify that any quoted upstream comment IDs still resolve.
- [ ] Pick one issue at a time. Allow 24-48h between posts.
- [ ] Watch for maintainer reactions; do not reply combatively.
- [ ] If a maintainer asks for collaboration, prefer issue / PR over
      thread chatter.

## Channels not in this batch

These have separate timing and require their own drafts:

- pkg.go.dev landing — automatic on tag push; no action.
- awesome-go / awesome-networking PR — short, schema-driven; not
  drafted here.
- HackerNews "Show HN" / r/golang — wait until at least one upstream
  reaction lands. Pre-mature posting before signal looks like spam.
