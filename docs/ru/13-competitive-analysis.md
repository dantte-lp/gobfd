### Программные реализации

#### FRR bfdd

[FRR](https://frrouting.org/) (Free Range Routing) — де-факто стандартный open-source маршрутизационный стек. Его BFD-демон (`bfdd`) — наиболее широко развёрнутая программная реализация BFD.

**Дефолты**: 300 мс TX/RX, 50 мс echo TX/RX, detect multiplier 3.

**Масштабирование**: SONiC (Azure/Microsoft) валидировал **64 BFD-сессии** с detection time 300 мс на commodity hardware. Это документированный потолок для их software-деплоя. У FRR нет жёсткого лимита сессий — 64 это просто порог, протестированный SONiC.

**Архитектура**: Однопоточная. BFD и BGP конкурируют за один процесс. При нагрузке на CPU (например, обработка 890K BGP-маршрутов на route reflector) `bgpd` занимает 100% одного ядра на 1-2 секунды, вызывая flapping BFD-сессий (FRR GitHub Issue #9078).

**Обработка пакетов**: FRR пересобирает BFD-пакет при каждой отправке. SONiC HLD явно указывает на эту неэффективность: «В текущей реализации FRR BFD-пакет пересобирается при каждой отправке — это overhead, учитывая что BFD должен отправлять пакеты каждые несколько миллисекунд». Рекомендация — кэшировать пакет в памяти и переиспользовать.

**Аппаратное ускорение**: Поддерживает distributed BFD data plane (`--dplaneaddr`), позволяющий вынести TX/RX в hardware/ASIC, но только на вендорских платформах.

#### BIRD2/BIRD3

[BIRD](https://bird.network.cz/) — легковесный маршрутизационный демон, популярный в IXP и исследовательских окружениях.

**Поддержка RFC**: RFC 5880/5881/5883/5882. Нет echo mode, нет demand mode.

**Бит CPI**: Бит Control Plane Independent **не выставляется** в BFD-пакетах (подтверждено tcpdump-анализом Vincent Bernat). Это означает, что BFD-сессии страдают при нагрузке на CPU BIRD.

**Масштабирование**: Публичных бенчмарков масштабирования нет. Используется в production у Wikimedia Foundation (с Juniper MX) и в окружениях Cilium/Kubernetes с интервалами 300 мс.

#### aiobfd

[aiobfd](https://github.com/network-tools/aiobfd) — Python asyncio реализация BFD.

**Достигнутые таймеры**: Успешно протестирована с **интервалами 10 мс** при работе с аппаратными маршрутизаторами. Для software-to-software требуется ~300 мс detection time.

**Ограничения**:
- GIL (Global Interpreter Lock) ограничивает истинный параллелизм
- Джиттер event loop asyncio: ~1-15 мс точность таймера
- Нет echo mode, нет аутентификации, нет demand mode
- Не тестировалась на масштабирование — вероятно, ограничена десятками сессий при агрессивных таймерах

#### Реализации на Go

**Thoro/bfd** (8 звёзд): gRPC-интерфейс для мониторинга. Известные баги: panic при обновлении таймеров во время активной сессии. Только simple password authentication. Нет бенчмарков.

**jthurman42/go-bfd**: Самоописание — «Experimental, incomplete and probably very wrong». Библиотека парсинга пакетов, не демон. Дефолтные таймеры 1 с. Поддерживает все типы аутентификации.

**google/gopacket**: Содержит BFD layer для парсинга пакетов, не реализация протокола.

**GoBFD**: Первая production-grade реализация BFD на Go с zero-allocation hot path, полным соответствием RFC 5880/5881, аутентификацией, echo mode, demand mode, поддержкой оверлеев (VXLAN/Geneve) и полным набором бенчмарков.

### Сравнение производительности

> Все числа GoBFD получены из микро-бенчмарков (`go test -bench`). Числа FRR и BIRD — из публичной документации, issue-трекеров и сторонних отчётов. Это **не** сравнительные тесты лоб-в-лоб — разные языки, архитектуры и методологии измерений делают прямое сравнение неточным. Мы приводим их для контекста, а не как окончательный рейтинг.

#### Обработка пакетов

| Метрика | GoBFD | FRR bfdd | aiobfd |
|---------|------:|----------|--------|
| Маршалинг пакета | 5.9 нс/оп, 0 аллокаций | Пересборка при каждой отправке (SONiC HLD) | Сериализация Python-объектов |
| Анмаршалинг пакета | 11.0 нс/оп, 0 аллокаций | C struct cast | Python-парсинг |
| Полный RX-путь | 64 нс/оп, 0 аллокаций | Не публикуется | Не публикуется |
| Полный TX-путь | 6.7 нс/оп, 0 аллокаций | Не публикуется | Не публикуется |

**Ключевой инсайт**: GoBFD использует кэшированные пре-собранные пакеты (копирование, не пересборка). FRR пересобирает BFD-пакет при каждой передаче. При 1000 сессий с интервалом 100 мс это 10 000 пересборок в секунду у FRR против 10 000 копирований памяти у GoBFD.

#### Масштабирование сессий

| Метрика | GoBFD | FRR bfdd (SONiC) | aiobfd |
|---------|------:|:----------------:|:------:|
| Максимум протестированных сессий | 1 000 (бенчмарк) | 64 (валидировано) | ~10 (оценка) |
| Демультиплексирование | O(1), 51 нс/оп | Линейный поиск | Python dict |
| Память на сессию | ~3 КБ (бенчмарк создания) | malloc/free на пакет | Python-объекты |
| Реконсиляция конфигурации | 94 мкс/15 сессий | Полный перезапуск | N/A |

#### Теоретическая пропускная способность

При 64 нс/оп полного пути приёма GoBFD теоретически может обрабатывать **15.6 миллионов пакетов в секунду** на одном ядре. При 6.7 нс/оп полного пути передачи теоретическая TX-пропускная способность — **149 миллионов пакетов в секунду**. Это числа микро-бенчмарков — реальная пропускная способность ниже из-за overhead системных вызовов, задержки сетевого стека и переключения контекста.

### Справка по аппаратным реализациям

Вендорские реализации BFD используют выделенное оборудование (ASIC/NPU) для обработки пакетов. Это **принципиально отличается** от программного BFD и не сравнимо напрямую. Мы приводим эту таблицу как справочную точку для понимания ландшафта BFD.

| Платформа | Макс. сессий | Мин. таймер | Аппаратное ускорение |
|-----------|-------------:|------------:|:--------------------:|
| Cisco ASR 9000 (NPU) | 8 000/LC при 50 мс; 600/LC при 3.3 мс | 3.3 мс | NPU, 7 фиксированных интервалов |
| Cisco 8000 (Silicon One) | 10 000 (VXLAN MH BFD) | Зависит от платформы | Да |
| Cisco NCS 5500 | 1 000/узел | 15 мс (до 100 сессий) | OAMP engine |
| Juniper MX (inline BFD) | По запросу | 10 мс (distributed), 100 мс (centralized) | ASIC firmware |
| Nokia SR Linux | 1 152 (8-слотовое шасси) | Настраивается | Аппаратный |
| Nokia 7750 SR | 250/IOM + CPM-np | 10 мс (CPM-np) | CPM-np hardware |
| Arista EOS | 200/ASIC (с HW-ускорением) | 50 мс | По умолчанию на поддерживаемых платформах |

**Экспертный контекст** (Manav Bhatia, участник IETF BFD WG): «BFD сложен при масштабировании. Становится НАМНОГО хуже, когда добавляешь безопасность. Нужно не только обрабатывать пакеты с высокой скоростью, но и вычислять SHA или MD5 дайджест для каждого. Это становится трудно при масштабировании сессий даже с аппаратным ускорением.»

**Экспертный контекст** (Ivan Pepelnjak, ipSpace.net): «Агрессивные BFD-таймеры могут вызывать ложные срабатывания. Большинство платформ реализуют BFD программно — если процессы с более высоким приоритетом используют весь CPU, BFD голодает.»

### Production-таймеры

Справочная таблица конфигураций BFD-таймеров в реальных развёртываниях:

| Интервал | Множитель | Время обнаружения | Контекст | Достижимость в ПО |
|---------:|----------:|------------------:|----------|-------------------|
| 1000 мс | 5 | 5000 мс | Google Cloud Router по умолчанию | Тривиально |
| 300 мс | 3 | 900 мс | FRR по умолчанию, NANOG «safe value», Juniper RE | Стандартно, надёжно |
| 100 мс | 3 | 300 мс | Цель SONiC, Juniper distributed | Достижимо в Go с изоляцией CPU |
| 50 мс | 3 | 150 мс | Минимум HW Cisco/Arista | Требует тщательной оптимизации |
| 10 мс | 3 | 30 мс | aiobfd достигла с HW peer | Теоретически возможно, нужен RT-тюнинг |

**Цель GoBFD**: интервалы 100 мс со стабильным detection time на commodity hardware, что ставит его между программным стандартом (300 мс) и аппаратным минимумом (10-50 мс).

**Формула detection time** (RFC 5880): `DetectionTime = DetectMult * max(RequiredMinRxInterval, remote DesiredMinTxInterval)`. До 25% случайного jitter ДОЛЖНО добавляться к TX-интервалам (секция 6.8.7). Дополнительные 10% jitter при DetectMult = 1.

### Методология и честное раскрытие

**Подход к измерениям**:

- Все числа GoBFD получены из Go микро-бенчмарков (`go test -bench -benchmem -count=6`)
- Статистический анализ через `benchstat` (медиана, 95% доверительные интервалы)
- Числа FRR и BIRD — из публичной документации, GitHub issue-трекеров и вендорских HLD
- Числа aiobfd — из README и тестовых отчётов
- Аппаратные числа — из вендорских даташитов и руководств по конфигурации

**Что мы НЕ утверждаем**:

- Мы не проводили сравнительных тестов лоб-в-лоб между GoBFD и FRR/BIRD/aiobfd
- Пропускная способность микро-бенчмарков (нс/оп) не переводится напрямую в системную производительность
- Разные языки (Go vs C vs Python) имеют принципиально разные характеристики производительности
- Измерения памяти из бенчмарков (Б/оп) — это «за операцию», а не общий footprint демона

**Воспроизводимость**:

Все бенчмарки GoBFD можно воспроизвести:

```bash
# Требуется только Podman — локальный Go не нужен
make up
make benchmark-save BENCH_VERSION=v0.4.0
cat testdata/benchmarks/v0.4.0/meta.json    # коммит, версия Go, дата
```

Мы приветствуем независимую верификацию и исправления к любым утверждениям в этом документе.
